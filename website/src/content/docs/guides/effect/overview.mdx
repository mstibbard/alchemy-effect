Effect programs in Alchemy express infrastructure and runtime code as a
single, connected program. If you haven't read [Effect vs Async](/concepts/effect-vs-async)
yet, start there for the high-level mental model.

## The two-phase pattern

Every Effect program runs in two phases:

1. **Init** (deploy time) — create resources, bind capabilities, wire
   event sources. This code runs once when you deploy.
2. **Runtime** (per request) — handle requests, process events, call
   bound capabilities. This code runs on every invocation.

```typescript
Effect.gen(function* () {
  // Phase 1: init — runs at deploy time
  const table = yield* DynamoDB.Table("UsersTable", {
    partitionKey: "pk",
    attributes: { pk: "S" },
  });
  const getItem = yield* DynamoDB.GetItem.bind(table);
  const putItem = yield* DynamoDB.PutItem.bind(table);

  return {
    // Phase 2: runtime — runs per request
    fetch: Effect.gen(function* () {
      const result = yield* getItem({ Key: { pk: { S: "user#1" } } });
      return yield* HttpServerResponse.json(result.Item);
    }),
  };
});
```

## What `bind` does

Calling `bind` on a capability does two things simultaneously:

1. **Deploy time** — attaches an IAM policy (AWS) or native binding
   (Cloudflare) to the function, granting exactly the permissions
   needed.
2. **Runtime** — returns a typed SDK wrapper that you call in your
   handlers.

This is the core mechanic that makes Effect programs fully connected:
the type system ensures you can't use a capability you didn't bind,
and the permissions are always in sync with your code.
