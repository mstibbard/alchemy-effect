Every alchemy-effect project has an `alchemy.run.ts` file that exports
a default Stack. The CLI provides three commands to manage its
lifecycle: **plan** to preview changes, **deploy** to apply them, and
**destroy** to tear everything down.

```typescript
// alchemy.run.ts
import * as Alchemy from "alchemy-effect";
import * as Cloudflare from "alchemy-effect/Cloudflare";
import * as Effect from "effect/Effect";

export default Alchemy.Stack(
  "MyApp",
  { providers: Cloudflare.providers() },
  Effect.gen(function* () {
    const bucket = yield* Cloudflare.R2Bucket("Bucket");
    const api = yield* Cloudflare.Worker("Api", {
      main: "./src/api.ts",
      bindings: { Bucket: bucket },
    });
    return { url: api.url };
  }),
);
```

## Plan

`alchemy plan` diffs the desired state against the current state and
prints what would change — without actually applying anything. It is
equivalent to `alchemy deploy --dry-run`.

```sh
alchemy plan
```

<Terminal content={planOutput} />

The plan uses `+` for creates, `~` for updates, and `-` for deletes.
No approval prompt is shown and no changes are made.

| Flag | Description |
| --- | --- |
| `--stage <name>` | Stage to plan against (defaults to `dev_$USER`) |
| `--env-file <path>` | Load environment variables from a file |

## Deploy

`alchemy deploy` computes the plan, asks for approval, and then
creates, updates, or deletes resources to match the desired state.

```sh
alchemy deploy
```

<Terminal content={deployOutput} />

On subsequent deploys, only changed resources are updated:

<Terminal content={updateOutput} />

| Flag | Description |
| --- | --- |
| `--stage <name>` | Stage to deploy to (defaults to `dev_$USER`) |
| `--yes` | Skip the approval prompt |
| `--dry-run` | Show the plan without applying (same as `alchemy plan`) |
| `--force` | Force updates for resources that would otherwise no-op |
| `--env-file <path>` | Load environment variables from a file |

## Destroy

`alchemy destroy` deletes every resource in the stack. It computes a
plan where all existing resources are marked for deletion, asks for
approval, and then removes them in dependency order.

```sh
alchemy destroy
```

<Terminal content={destroyOutput} />

| Flag | Description |
| --- | --- |
| `--stage <name>` | Stage to destroy (defaults to `dev_$USER`) |
| `--yes` | Skip the approval prompt |
| `--dry-run` | Show what would be deleted without actually deleting |
| `--env-file <path>` | Load environment variables from a file |

## Stages

Every deploy targets a **stage** — an isolated instance of your stack
like `dev`, `prod`, or `dev-sam`. The stage defaults to `dev_$USER`
(e.g. `dev_sam`) so each developer gets their own environment.

```sh
alchemy deploy --stage prod
alchemy destroy --stage dev_sam
```

Resources are namespaced by stage, so `dev` and `prod` deployments
never interfere with each other. Physical names include the stage
(e.g. `myapp-dev-bucket-abc123`).

## State

Alchemy persists resource state between deploys in the `.alchemy/`
directory so it can diff desired state against current state and apply
only the necessary changes. Add `.alchemy/` to your `.gitignore` for
local development. For team use, configure a remote state store.
